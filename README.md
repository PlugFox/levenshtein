# Levenshtein на Dart: восемь оптимизаций и замеры

Воспроизведение алгоритма Дамерау–Левенштейна из статьи
[«Прокачиваем локальный поиск на Dart и Flutter»](https://habr.com/ru/articles/1031212/),
последовательная цепочка оптимизаций (v1 → v8) и подробные бенчмарки на
синтетическом корпусе и реальном словаре русского языка
([danakt/russian-words](https://github.com/danakt/russian-words),
1 532 628 слов).

## Среда замеров

| Параметр       | Значение                                                                |
| -------------- | ----------------------------------------------------------------------- |
| Дата           | 2026-05-06                                                              |
| CPU            | 13th Gen Intel Core i7-13700K (16 cores / 24 threads, до 5400 MHz)      |
| RAM            | 31 GiB                                                                  |
| ОС             | Linux 7.0.2-2-cachyos (CachyOS), x86_64, ядро собрано clang 22.1.3      |
| Dart SDK       | 3.11.4 (stable, 2026-03-24, `linux_x64`)                                |
| JIT            | `dart run benchmark/<file>.dart` (warmup harness'ом)                    |
| AOT            | `dart compile exe benchmark/<file>.dart -o /tmp/<bin> && /tmp/<bin>`    |
| Tooling        | `package:benchmark_harness` 2.3.x (RunTime metric, µs/итерация `run()`) |

`benchmark_harness` сам делает прогрев + измеряет среднее по нескольким
итерациям и возвращает µs за **один** `run()`. Каждый `run()` в этом
проекте делает либо 200 пар (синтетический), либо 50 запросов × 5000
кандидатов = 250 000 distance-вычислений (top-K).

## Что в каждой версии

| Версия               | Идея                                                                                                                                                      | Семантика            | API                                          |
| -------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------- | -------------------- | -------------------------------------------- |
| **v1 baseline**      | Дословно из статьи: две `List<int>`, `min` из `dart:math`, `String.codeUnitAt`. Особенность статьи — двухстрочная DP с упрощённой транспозицией (см. ниже) | Дамерау-«как-в-статье» | `damerauLevenshteinDistance(a, b)`           |
| **v2 typed**         | + `Uint16List` вместо `List<int>`, code units строк закэшированы один раз через `Uint16List.fromList(s.codeUnits)`                                        | =                    | `damerauLevenshteinV2(a, b)`                 |
| **v3 trim**          | + срезка общего префикса/суффикса, единый `Uint16List(2*(n+1))` с переключением offset, inline `min` через тернарник                                      | =                    | `damerauLevenshteinV3(a, b)`                 |
| **v4 threshold**     | + `int? threshold`: prefilter `\|len1−len2\| > k`, per-row early-exit, `Uint32List` для безопасной арифметики                                               | =                    | `damerauLevenshteinV4(a, b, threshold: k)`   |
| **v5 Myers**         | Bit-parallel Myers'99: O(n) машинных слов для `m ≤ 64`. Для `m > 64` — fallback в v3                                                                      | Классический Левенштейн | `levenshteinMyers(a, b)`                     |
| **v6 SIMD**          | Two-pass DP с `Float32x4.min` (4 ячейки за такт), скалярный sweep для insert                                                                              | Классический Левенштейн | `levenshteinSimd(a, b)`                      |
| **v7 optimal**       | Best-of: v2 (typed) + v3 (trim) + v4 (threshold/early-exit) + **pre-allocated module-level буферы** (`_u1, _u2, _prev, _curr`), unrolled edge-cases       | Дамерау-«как-в-статье» | `damerauLevenshteinV7(a, b, threshold: k)`   |
| **v8 bag-filter**    | + bag-of-chars prefilter: оценка `Lev ≥ ⌈\|M1 ⊕ M2\|/2⌉` через мультимножество code units; epoch-based reset; **двусторонний early-exit** (success/failure) | Дамерау-«как-в-статье» | `damerauLevenshteinV8(a, b, threshold: k)`   |

> Транспозиция в статье использует `prevRow[j-2]` (то есть `d[i-1][j-2]`)
> вместо канонического `d[i-2][j-2]` — это «двухстрочная Damerau», которая
> детектирует не все транспозиции (например, `("ab","ba")` даёт **2**, а не
> 1). Все версии v1-v4, v7, v8 воспроизводят эту особенность для прямой
> сравнимости. Для канонического OSA-Дамерау нужна третья DP-строка.

### Что НЕ реализовано (намеренно)

- **Bit-parallel Damerau (Hyyrö'03)** — добавляет асимметричную обработку
  HP/HN с переносами, выходит за объём задачи. В v5 транспозиции нет.
- **Block-Myers** для `m > 64` — нетривиальная multi-precision
  carry-арифметика с разной обработкой positive/negative переносов между
  блоками. v5 для `m > 64` делегирует в v3.
- **Полный Ukkonen-band** — диагональная полоса с INF-обивкой границ
  оказалась хрупкой и на коротких строках (наш сценарий) не окупается.
  Сохранили только threshold-prefilter и per-row early-exit (это и даёт
  основной выигрыш).

## Замер 1: синтетический корпус

`benchmark/levenshtein_benchmark.dart`. Каждый корпус = 200 пар
(`Random(42)`-генератор, половина — мутации с 1-3 правками,
половина — случайные строки). `run()` считает дистанцию по всем 200 парам.

Метрика — µs за один `run()`. Для перевода в µs/пара делите на 200.

### Ключевые наблюдения по синтетике

- **На short** (5–15 символов) выигрывает **v8** с bag-filter'ом: `~110 µs`
  AOT, что в **×2.0 раз** быстрее `v4_threshold` и в **×24 раза** быстрее
  baseline'а v1.
- **На medium/long** bag-filter перестаёт окупаться (его стоимость
  `O(n1+n2)` сравнивается с per-row early-exit) — здесь немного
  выигрывает **v7**.
- **Myers** силён на medium (3 ms vs 9 ms у v2/v3 в JIT) — bit-parallel
  даёт ~×3 на длинах, помещающихся в одно 64-битное слово.
- **SIMD-эмуляция через Float32x4** не окупается ни на одном корпусе —
  накладные расходы на lane-wise gather превышают выигрыш от
  параллельного `min`. Реализация ценна как иллюстрация подхода.
- **AOT часто медленнее JIT** на baseline-вариантах, потому что JIT
  специализирует горячие пути. Но на оптимизированных версиях AOT
  и JIT идут вровень.

## Замер 2: top-K fuzzy search на реальном словаре

`benchmark/topk_search_benchmark.dart`. Сценарий статьи (адресный поиск):
- haystack: 5000 случайных слов из `russian.txt` (длина 4–20).
- 50 запросов: каждый = случайное слово из haystack с 1-3 правками
  (insert / delete / substitute / transpose).
- Для каждого запроса проходим **весь** haystack, считаем Левенштейн,
  держим top-5 по дистанции.
- `run()` = 50 запросов × 5000 кандидатов = **250 000 distance-вычислений**.

### Результаты (µs / 250 000 операций)

| Вариант             |        JIT |        AOT |    × v1 (AOT) | × v4_thr (AOT) |
| ------------------- | ---------: | ---------: | ------------: | -------------: |
| v1_baseline         | 2 014 417  | 4 069 969  |          ×1.0 |              — |
| v2_typed            |   900 232  | 1 151 212  |          ×3.5 |              — |
| v3_trim             | 1 470 460  | 1 563 843  |          ×2.6 |              — |
| v7_optimal (no thr) | 1 419 924  | 1 343 499  |          ×3.0 |              — |
| v5_myers            | 1 304 065  | 1 116 241  |          ×3.6 |              — |
| **v4 threshold=2**  |   305 100  |   284 172  |        ×14.3  |          ×1.00 |
| **v7 threshold=2**  |   196 302  |   179 678  |        ×22.7  |          ×1.58 |
| **v8 threshold=2**  | **107 409**| **108 139**|       **×37.6** |       **×2.63** |

### Ключевые наблюдения по top-K

- **v8 уверенно побеждает** на реальных данных: AOT выдаёт **108 ms** на
  250К операций (≈ **0.43 µs/distance**) — в **2.6 раза** быстрее
  `v4_threshold` и в **37 раз** быстрее baseline'а из статьи.
- **v3 проигрывает v2** — на случайных русских словах общий префикс/
  суффикс встречается редко, обвязка trim'а не окупается. Это
  «удивительный» результат, который видно только на реальных данных:
  на синтетическом корпусе с мутациями v3 вёл себя лучше v2.
- **Myers** проигрывает даже v2 — медианная длина слова ~6 символов,
  и стоимость построения `Map<int,int>` для bitmask дороже самой DP
  на таких коротких строках.
- На JIT и AOT расстановка идентична. Разница между ними < 5%.

## Что дало главный выигрыш

В порядке вклада в финальную скорость v8 vs v1 (×37):

1. **Threshold prefilter + per-row early-exit (v4)** — самый большой
   единичный множитель: **×14**. Большинство кандидатов из haystack
   заведомо далеки от запроса; их можно отбраковать после 1-2 строк
   DP без полного расчёта.

2. **Pre-allocated module-level буферы (v7)** — **×1.6** поверх v4.
   Каждый из 250К вызовов перестал делать 3 аллокации Uint16List.

3. **Развёрнутые edge-case итерации (v7)** — i=1 и j=1 идут отдельными
   блоками, во внутреннем горячем цикле нет проверок `i > 1 && j > 1`,
   что помогает предсказателю ветвлений.

4. **Bag-of-chars prefilter (v8)** — **×1.7** поверх v7 на коротких
   строках. Дешёвая оценка `Lev ≥ ⌈|M1 ⊕ M2|/2⌉` через мультимножество
   code units, с двусторонним early-exit (success при достижении нужного
   intersection и failure при переборе nonMatch).

## Где v8 не оптимален

- **На длинных строках (≥ 50 символов)** bag-filter стоит `O(n1+n2)`
  и не окупается, потому что per-row early-exit и так быстр. Здесь
  на ~7-15% лучше **v7** (см. medium/long в синтетическом замере).
- В production имеет смысл выбирать вариант по доменy:
  - короткие токены (адреса, имена, ID, коды) → **v8**,
  - длинные документы → **v7** (или **v5 Myers** при `m ≤ 64` без
    транспозиции).
- Для единого варианта можно добавить дешёвый gate в v8:
  `if (len1 + len2 > 40) skip bag` — тогда он совпадёт с v7 на длинных.

## Корректность

`dart test` запускает 27 тестов:
- фиксированные пары (`kitten`/`sitting`=3, пустые, идентичные и т.п.);
- 1000 случайных пар, каждый вариант сверяется с эталонной реализацией
  (отдельный `referenceArticle` для Damerau-«как-в-статье»,
  `referenceLevenshtein` для классического);
- кросс-проверка v2…v8 против v1 на 1000 случайных пар каждая;
- v8 с разными threshold (0, 1, 2, 3, 5, 10) сверяется с v7;
- bag-filter протестирован на кириллице (включая `('абвгд', 'едгва')`
  с полной перестановкой символов — bag-filter не должен ложно
  отвергать такие пары).

Все 27 тестов проходят на JIT.

## Что бы ещё попробовать

- **Block-Myers (Hyyrö'03)** для `m > 64` — корректная multi-precision
  carry-арифметика. Дала бы ×10+ для длинных строк.
- **Bit-parallel Damerau** (тоже Hyyrö) — Myers с поддержкой транспозиции.
- **Length-gated bag-filter в v8** — тривиальное `if (len1+len2 > 40)
  skip bag` сделает v8 универсально не хуже v7.
- **FFI к Rust/C** — финальная оптимизация при необходимости. Не пробовали:
  пользователь хотел оптимизации в чистом Dart.
- **SIMD через anti-diagonal DP** — теоретически работает на длинных
  строках, но требует серьёзной перестройки индексирования. Float32x4
  с гарантированным маппингом в SSE/NEON через `dart:typed_data`
  поддерживает все нужные операции.
